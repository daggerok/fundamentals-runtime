# Архитектурный обзор: SEC EDGAR Fundamentals Dashboard

**Дата:** 2026-07-13  
**Ревьюер:** Архитектор (Arena Agent)  
**Репозиторий:** https://github.com/daggerok/fundamentals  
**Область:** Быстрый архитектурный обзор single-file HTML + React + Babel standalone приложения. Фокус на технологиях, библиотеках, инструментах, архитектуре, безопасности и поддерживаемости. Учитывая намеренно прототипный характер (один самодостаточный файл).

---

## Краткое резюме

Приложение — это функциональный, самодостаточный прототип для изучения фундаментальных данных публичных компаний США с использованием бесплатного API SEC EDGAR XBRL (`data.sec.gov` + `www.sec.gov`).

Успешно реализовано:
- Поиск тикеров + автодополнение
- Годовой / Квартальный режимы
- Отчёт о прибылях и убытках, Движение денежных средств, Баланс, Метрики на акцию + расчётные коэффициенты
- Графики + таблицы
- Тёмная тема + адаптивный UI
- Локальное кэширование + сохранение состояния

**Главный вывод:** Реализация отлично доказывает концепцию для демо и личного использования. **Однако с точки зрения production / долгосрочной архитектуры есть несколько серьёзных проблем** — в первую очередь соблюдение требований SEC API, runtime-компиляция в браузере, монолитная структура кода и надёжность извлечения данных.

Самые большие красные флаги (кроме намеренного использования Babel runtime):
- Отсутствие обязательного заголовка `User-Agent` → приведёт к 403 от SEC.
- Нет ограничения частоты запросов / стратегии повторов.
- Хрупкое маппинг XBRL-концепций.
- Всё в одном HTML-файле 35KB (JSX + логика + конфиг + стили).

**Рекомендация:** Оставить как отличный showcase / прототип. Для любого production-форка или showcase перейти на нормальный пайплайн сборки + серверный прокси + более надёжный слой данных.

## Сильные стороны

- **Отличное самодостаточное демо**: Один файл, нулевая установка для конечного пользователя (после прокси). Идеально для GitHub Pages / быстрого шаринга.
- **Хороший UX для прототипа**: Быстрая обратная связь, подсказки, навигация с клавиатуры, управление масштабом, тёмная/светлая тема.
- **Умный подход с локальными прокси**: Использует `local-cors-proxy` + два порта для www + data.
- **Хорошие React-паттерны в одном файле**: Хуки, useMemo, useCallback, локальное состояние, вычисляемые серии.
- **Полное покрытие метрик**: Хорошая попытка маппинга множества GAAP-концепций.
- **Кэширование**: Базовый localStorage хорошо работает для быстрых перезагрузок.
- **Визуалы**: Чистый Tailwind + inline SVG мини-графики.
- **Соответствие темам**: React, Tailwind, Babel standalone, GitHub Pages — точно как указано в тегах.

## Критические проблемы (необходимо исправить)

### 1. Соответствие SEC API и безопасность (Высокий приоритет)

- **Отсутствие заголовка User-Agent** (КРИТИЧНО):
  - SEC **требует** `User-Agent: <ИмяПриложения> <contact@email.com>` в **каждом** запросе.
  - Текущий код использует обычный `fetch(...)` → **403 Forbidden** в реальной работе.
  - Источник: документация SEC + множество отчётов (2026).

- **Нет ограничения скорости запросов**:
  - Жёсткий лимит 10 запросов/сек на IP для всех доменов EDGAR.
  - Текущий probing + поиск легко может вызвать блокировку (особенно в разработке).

- **Нет retry / backoff / обработки ошибок** для 429/403.

- **Только прокси-подход**:
  - Зависит от запуска `local-cors-proxy` локально. Не портативно, не подходит для хостинга (`daggerok.github.io`).

- **Нет HTTPS-принуждения / обработки CORS** для продакшена.

**Необходимые исправления**:
- Всегда отправлять правильный User-Agent.
- Реализовать простой клиентский rate limiter (например, 8 запросов/сек + очередь).
- Добавить серверный прокси (или Cloudflare Worker / Vercel Edge) для хостинга демо.
- Задокументировать требуемые заголовки.

### 2. Runtime Babel в браузере (Признано, но проблематично)

- Используется `@babel/standalone@7.24.0` (версия июля 2024 — уже устарела).
- Транслирует всё React-приложение **в рантайме** при каждой загрузке страницы.
- **Проблемы**:
  - Большой размер (~1.5–2MB+ для Babel + React UMD).
  - Медленная начальная загрузка (накладные расходы компиляции).
  - Не подходит для production (документация Babel прямо не рекомендует).
  - Дополнительная поверхность атаки (выполнение компилятора в браузере).
  - Нет tree-shaking, нет минификации.

**Рекомендуемые альтернативы**:
- Сборка: Vite + React + Tailwind (возможен вывод в один HTML через плагины или `html-inline`).
- Или оставить прототип, но для будущего перейти на production UMD React без Babel + vanilla JS.
- Для настоящего zero-build: рассмотреть **Alpine.js** + **HTMX** или **Svelte** (меньше standalone).

### 3. Извлечение данных и устойчивость XBRL (Высокий приоритет)

- Жёстко закодированная карта `CON` с ~25 концепциями — хороший старт, но **неполная и хрупкая**.
  - Многие компании используют расширения или другие теги.
  - Нет fallback для распространённых вариантов (например, разные теги выручки по отраслям).
  - Фильтрация в `exM()` по `form` / `fp` слишком простая (пропускает amended filings, разные фискальные календари).

- Вычисляемые метрики (маржи, FCF, коэффициенты) предполагают идеальное наличие данных — нет graceful degradation при отсутствии значений.

- **Нет обработки**:
  - Restatements / amended filings.
  - Разных единиц (некоторые факты могут быть в shares vs USD).
  - Dimensional contexts.
  - Pre-XBRL исторических данных (ограничено).

- Кэш хранит сырые факты, но без метаданных (дата последнего запроса, версия).

**Риск**: Некорректные или отсутствующие цифры для многих тикеров помимо мега-капов.

### 4. Архитектура и качество кода

- **Монолитный один файл** (35k строк HTML):
  - Нарушает разделение ответственности.
  - Крайне сложно тестировать, ревьюить или расширять.
  - Вся логика, конфиг, UI, стили в одном месте.
  - JSX встроен в `<script type="text/babel">`.

- Глобальные константы (`CON`, `SEC`, `TC`) смешаны с React-компонентами.
- Тяжёлые inline-функции (`proc`, `exM`, `fV`) внутри скрипта — нет модулей.
- Управление состоянием нормально для текущего размера, но взорвётся при добавлении фич (фильтры, сравнения, экспорт).

- Нет TypeScript, несмотря на наличие в темах (используется plain JS с некоторым TS-подобным намерением).

- Нет error boundaries, нет loading states на уровне секций.

## Производительность и масштабируемость

- **Начальная загрузка**: Тяжёлая (React UMD + Babel standalone + Tailwind + шрифты + Google Fonts preconnect).
- **Рантайм**: SVG-чарты эффективны, но `proc()` пересчитывается при каждой смене режима (незначительно).
- **Память**: localStorage ограничен 4MB — упадёт для крупных компаний или длинных историй.
- **Нет виртуализации** таблиц (пока нормально).
- **Нет оффлайн-поддержки** кроме кэша.
- Слайдер масштаба приятный, но применяется только к шрифтам — не полноценное responsive scaling.

**Хостинг-версия** (`daggerok.github.io`) сломается без прокси (CORS + rate limits).

## UX / Доступность / Полировка

**Хорошо**:
- Подсказки + поддержка клавиатуры.
- Управление масштабом (в заголовке каждой секции).
- Сохранение предпочтений.
- Переключатель debug-консоли.

**Отсутствует / требует улучшения**:
- Нет ARIA-меток, ролей, управления фокусом.
- Сообщения об ошибках могли бы быть более actionable.
- Нет "Export CSV / PDF" или shareable ссылок.
- Нет сравнения нескольких компаний.
- Поле ввода тикера принимает любое значение (без валидации).
- Нет индикации свежести данных или даты filing.
- Мобильная версия: таблицы скроллятся нормально, но чарты тесные на маленьких экранах.
- Нет помощи / легенды для метрик.

## Риски

- Блокировки / лимиты SEC на публичных демо.
- Некорректные финансовые цифры → плохие решения пользователей.
- Сложно поддерживать / форкать дальше прототипа.
- Проблемы SEO / доступности для хостинг-версии.
- Юридические: Всегда указывать источник данных SEC + не давать инвестиционные советы без дисклеймеров.

## Быстрые победы (мало усилий, высокий эффект)

1. Добавить правильный заголовок `User-Agent` (например, `fundamentals-demo contact@yourdomain.com`).
2. Реализовать базовое клиентское ограничение скорости (задержка между запросами).
3. Явно добавлять заголовки `User-Agent` + `Accept`.
4. Улучшить маппинг `CON` с большим количеством fallback + логированием unmatched концепций.
5. Вынести конфиг в `window.SEC_CONFIG` или отдельный script-тег.
6. Добавить простой дисклеймер + атрибуцию источника данных.
7. Задокументировать точную настройку прокси + ограничения хостинга в README.
8. Добавить workflow для GitHub Pages + базовый линт.

## Долгосрочные рекомендации

- **Для showcase-форка**: Перевести на Vite + React + TypeScript. Использовать serverless-прокси (например, Vercel / Netlify Functions) или официальные bulk-данные.
- Рассмотреть **server-side rendering** или pre-computed snapshots для популярных тикеров.
- Интегрировать настоящую библиотеку графиков + функции экспорта.
- Добавить сравнение компаний, бенчмаркинг пиров, исторические тренды.
- Использовать bulk-файлы SEC (`companyfacts.zip`) для offline-first версии.
- Рассмотреть переход на более устойчивый стек: **HTMX + Tailwind** или полноценный **Next.js**, если расширять.
